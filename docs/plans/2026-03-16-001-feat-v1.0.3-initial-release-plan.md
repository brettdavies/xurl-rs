# feat: v1.0.3 Initial Release

## Overview

Ship the initial public release of xurl-rs v1.0.3 -- the Rust port of Go xurl. The codebase is fundamentally ready (137
tests passing, clean formatting, crate packages). This plan covers the final polish: fix 3 code issues, wire up Homebrew
automation, set repo metadata, and execute a 10-step release sequence.

## Problem Statement / Motivation

The port is code-complete but has 3 issues found during review that would either break CI on main (`-Dwarnings`) or
violate established patterns (OutputConfig, American English). Additionally, the release pipeline is 90% built -- the
`repository_dispatch` to homebrew-tap is documented in RELEASING.md but not yet implemented in `release.yml`.

## Proposed Solution

Fix all issues on `dev`, wire up the full release automation, then execute the ordered release sequence: merge, go
public, tag, release, verify. (See brainstorm: `docs/brainstorms/2026-03-16-initial-release-brainstorm.md` for decision
rationale.)

## Implementation

### Phase 1: Code Fixes on `dev`

Three fixes, all on `dev` branch. Commit individually per SRP.

#### 1a. Fix Clippy Warnings in `tests/conformance/mod.rs`

Two warnings that will fail CI on main (`RUSTFLAGS="-Dwarnings"` in `ci.yml`):

| Warning | Location | Fix |
|---------|----------|-----|
| Redundant field name | `tests/conformance/mod.rs:198` | `port_exit: port_exit` -> `port_exit` |
| Dead code (`tags` field) | `tests/conformance/mod.rs:49` | Add `#[allow(dead_code)]` to the `tags` field (it's deserialized from TOML but unused in logic) |

**Verify:** `RUSTFLAGS="-Dwarnings" cargo clippy --all-targets` must produce zero warnings.

#### 1b. Fix `println!`/`process::exit` Bypass in `src/cli/commands/mod.rs:36-39`

**Problem:** `run_raw_mode()` uses naked `println!()` + `process::exit(1)` when no URL is provided. This bypasses
OutputConfig (breaks `--output json` and `--quiet`) and the structured exit code system in `main.rs:41-54`.

**Current code:**

```rust
} else {
    println!("No URL provided");
    println!("Usage: xr [OPTIONS] [URL] [COMMAND]");
    println!("Try 'xr --help' for more information.");
    std::process::exit(1);
};
```

**Fix:** Return an error through the normal error path:

```rust
} else {
    return Err(XurlError::Api(
        "No URL provided. Usage: xr [OPTIONS] [URL] [COMMAND]. Try 'xr --help' for more information.".to_string(),
    ));
};
```

This flows through `main.rs:32-36` -> `exit_code_for_error()` -> `out.print_error()`, respecting `--output json` and
`--quiet`. The `XurlError::Api` variant is appropriate here -- it's the general-purpose error variant used throughout
the codebase for user-facing messages (see `src/error.rs`). The error maps to `EXIT_GENERAL_ERROR (1)`, matching the
original `process::exit(1)`.

**Pattern reference:** The solutions doc at `docs/solutions/rust-patterns/rust-cli-port-methodology.md` explicitly
states "no naked `println!` in command code."

#### 1c. Fix British Spelling in `src/api/shortcuts.rs:54`

**Change:** `/// Normalises a username` -> `/// Normalizes a username`

Consistency with American English used throughout the codebase (e.g., `normalize_output` in
`tests/conformance/normalize.rs`).

#### 1d. Commit RELEASING.md Changes

The working tree has uncommitted modifications to `RELEASING.md` that document the `repository_dispatch` pattern and
`HOMEBREW_TAP_TOKEN` secret. Commit these as part of the infrastructure work.

**WARNING:** `docs/brainstorms/` is untracked in the working tree. It MUST NOT be staged or committed to `dev`. The
`guard-main-docs.yml` workflow blocks `docs/brainstorms/` changes from merging to main and will fail the PR.

### Phase 2: Release Pipeline Automation

#### 2a. Add `repository_dispatch` Job to `release.yml`

**File:** `.github/workflows/release.yml`

Add a `homebrew` job after the existing `release` job. Uses `gh api` (pre-installed on all GitHub-hosted runners) with
inline `v` prefix stripping (see brainstorm: decision #4).

```yaml
  homebrew:
    needs: release
    runs-on: ubuntu-latest
    steps:
      - name: Dispatch Homebrew formula update
        env:
          GH_TOKEN: ${{ secrets.HOMEBREW_TAP_TOKEN }}
          VERSION: ${{ github.ref_name }}
        run: |
          gh api repos/brettdavies/homebrew-tap/dispatches \
            --method POST \
            -f event_type=update-formula \
            -f 'client_payload[formula]=xurl-rs' \
            -f "client_payload[version]=${VERSION#v}" \
            -f 'client_payload[repo]=brettdavies/xurl-rs'
```

Key details:

- `needs: release` -- fires after the GitHub Release is created (tarball already available from tag push)
- `${VERSION#v}` strips the `v` prefix so homebrew-tap receives `1.0.3`, not `v1.0.3`
- `gh api` handles auth via `GH_TOKEN` env var, returns exit code 1 on HTTP errors with error body on stdout
- Bracket notation (`client_payload[key]=value`) creates nested JSON without manual string building
- No third-party action dependency (`peter-evans/repository-dispatch`) and no manual JSON/header construction (raw
  `curl`)

#### 2b. Create `update-formula.yml` and `tests.yml` in `homebrew-tap`

**Files:**

- `/home/brett/dev/homebrew-tap/.github/workflows/update-formula.yml`
- `/home/brett/dev/homebrew-tap/.github/workflows/tests.yml`

Follow the existing plan at `homebrew-tap/docs/plans/2026-03-16-automated-formula-updates-plan.md` exactly.

**`update-formula.yml`** -- receives dispatch events, downloads tarball, computes sha256, updates formula via `sed`,
  commits and pushes. Generic (payload-driven, not hardcoded to xurl-rs) and idempotent (`git diff --cached --quiet`
  exits cleanly if no changes). Uses unconditional `sed` replace for both url and sha256 (sha256 pre-seeded in Task 0,
  no conditional insert logic).

**`tests.yml`** -- the official `brew test-bot` CI workflow from the `brew tap-new` template. Runs on pushes to main and
  PRs. When `update-formula.yml` pushes a formula change, this workflow automatically validates it via `brew test-bot
  --only-tap-syntax` (runs `brew audit`, `brew style`, and `brew readall` against the entire tap). Uses
  `Homebrew/actions/setup-homebrew` (not `actions/checkout` + `brew tap`) to correctly symlink the repo into the
  Homebrew tap directory.

### Phase 7: Tag and Release

```bash
git checkout main && git pull
git tag v1.0.3
git push origin v1.0.3
```

This triggers `release.yml`:

1. **build** -- 5 target matrix (linux x86_64/aarch64, macOS x86_64/aarch64, Windows x86_64)
2. **release** (needs: build) -- downloads artifacts, creates GitHub Release with auto-generated notes
3. **publish** (needs: build, parallel with release) -- `cargo publish` to crates.io
4. **homebrew** (needs: release) -- dispatches `update-formula` to homebrew-tap

### Phase 8: Verify All 4 Distribution Channels

Manual verification checklist:

- [ ] **GitHub Release:** `gh release view v1.0.3` -- 5 binaries attached, release notes present
- [ ] **crates.io:** `cargo install xurl-rs && xr --version` -- prints `xurl 1.0.3`
- [ ] **Homebrew:** `brew tap brettdavies/tap && brew install xurl-rs && xr --version` -- prints `xurl 1.0.3`
- [ ] **Homebrew formula updated:** Check homebrew-tap repo for automated commit with sha256
- [ ] **From source:** `git clone https://github.com/brettdavies/xurl-rs && cd xurl-rs && cargo build --release &&
  ./target/release/xr --version`

## Technical Considerations

**crates.io is permanent.** Once `cargo publish` succeeds, version 1.0.3 is consumed forever. `cargo yank` only prevents
  new downloads -- the version number cannot be reused. If a critical bug is found post-publish, the fix goes into
  1.0.4.

**Partial failure.** The `publish` and `release` jobs run in parallel after `build`. If one succeeds and the other
  fails:

- *Release fails, publish succeeds:* Manually create the GitHub Release and re-trigger the homebrew dispatch
- *Publish fails, release succeeds:* Fix the issue and re-run `cargo publish` manually (the version hasn't been
  consumed)

**Changelog.** The `ci.yml` changelog job (git-cliff) fires on push to main but does not commit the output. The
  CHANGELOG.md from dev will be the static version on main. This is a known incomplete feature -- not blocking for
  v1.0.3.
